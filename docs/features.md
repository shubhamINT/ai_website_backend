# Feature List

Complete capability inventory for the Indus Net AI Website Backend.

---

## 1. Real-Time Voice Conversation

- Browser visitor speaks directly into the website — no app install, no form fill.
- Sarvam AI STT (`saaras:v3`, transcribe mode) converts speech to text.
- OpenAI `gpt-4.1` drives the conversation; turn detection via `MultilingualModel` + Silero VAD.
- Sarvam AI TTS (`bulbul:v3`, speaker `simran`) synthesises speech with a fixed Indian-English voice profile.
- Noise cancellation (`livekit-plugins-noise-cancellation`) available as an optional room input filter.
- Background ambient audio (office ambience + typing sound) plays during agent thinking time.
- `preemptive_generation=True` — agent starts generating before TTS output finishes.

---

## 2. Filler Phrase Generation

- While the user is speaking, agent fires filler phrases ("I see...", "Sure, let me check…") after a 2–3 s delay.
- Fillers repeat every 5–8 s if user keeps speaking.
- Each filler is a standalone `gpt-4o-mini` call with the last 4 conversation turns so tone matches context.
- Filler loop cancelled immediately when user stops speaking.

---

## 3. Silence Watchdog & Idle Shutdown

- **Silence watchdog:** Triggers if no user message arrives within a configured window. Agent prompts the user.
- **Idle shutdown controller:** Shuts down session if agent has been idle too long (no active speaking or processing state).
- Both controllers are non-blocking async tasks alongside the main session.

---

## 4. Company Knowledge Base Search

- Semantic similarity search over a ChromaDB vector store (`company_knowledge` collection).
- Embeddings via OpenAI `text-embedding-3-small`.
- Returns top-k markdown-formatted results back to the LLM for grounded answers.
- Vector store persisted locally at `src/services/vectordb/chroma_db/`.

---

## 5. Live Internet Search

- Three concurrent SearXNG searches per query: **general**, **news**, **IT/tech**.
- Query is auto-enriched (conversational fluff stripped) before each search.
- All three run in parallel via `asyncio.gather`.
- Results preprocessed and returned to the LLM as sectioned snippets `[General]`, `[News]`, `[Tech / IT]`.
- Same enriched query drives image search for flashcard visuals.

---

## 6. AI-Generated UI Cards

The agent shows **exactly one visual per turn**. Routing: a dedicated screen tool
(offices, forms, distance, meeting) if the intent fits → else an image-flashcard
deck (`publish_ui_stream`) for `render_image_flashcards` topics (case studies,
team/CEO, services showcase, company background) → else a text infographic
(`publish_infographic`).

### 6a. Image Flashcard Deck (`publish_ui_stream` → `ui.flashcard`)

- After answering, agent calls `publish_ui_stream` which streams JSON card payloads to the frontend over the `ui.flashcard` LiveKit data topic.
- Cards generated by `gpt-4o-mini` (`UIAgentFunctions`) in streaming mode — a dynamic-count deck (typically 1–6) appears one by one.
- Each card is either an image `flashcard` (resolves a media asset — mapped from `MEDIA_ASSETS` **or** fetched live via SearXNG image search) **or** a text `infographic` mixed into the deck where an image adds nothing (same payload as the standalone infographic card below).
- Stream ends with an `end_of_stream` marker so the frontend knows rendering is complete.

> `MEDIA_ASSETS` was trimmed — testimonial, case-study, partner, and `cloud_devops` image entries were removed; those topics now render as text infographics rather than images.

### 6b. Text Infographic (`publish_infographic` → `ui.infographic`)

- Standalone infographic card (NO images) authored directly by the main agent LLM — the default visual for substantive answers that are not `render_image_flashcards` topics: pricing, process, explainers, partners, comparisons, greetings, general Q&A.
- Payload (`type: infographic`): a composed card with an optional `hero` and ordered, typed section blocks (`markdown`, `bullet_list`, `icon_bullets`, `stats`, `cta_banner`), plus optional `chips`, `visual_intent` (`neutral|urgent|success|warning|processing`), and `icon` (Lucide name). The full schema lives in [`docs/frontend-infographic-contract.md`](frontend-infographic-contract.md).
- Holds richer, longer structured text than a terse image flashcard. Like the flashcard deck, it REPLACES everything on screen. Spoken reply stays plain prose (markdown is rendered visually only, never spoken).

---

## 7. Flashcard Memory & Recall

- Every image-flashcard batch is saved to **Mem0** (backed by a separate ChromaDB collection `chroma_db_mem0/`) keyed by `user_id`.
- `recall_and_republish_ui_content` retrieves previously shown **flashcards** from Mem0 and re-streams them with `recalled=true`. It does **not** replay `infographic` content — the agent re-authors those via `publish_infographic`.
- Mem0 failures are non-fatal — logged and the session continues.

---

## 8. Global Presence Panel

- `publish_global_presence` pushes an office location panel to the frontend over `ui.global_presense`.
- Triggered by conversation intent around global offices / worldwide presence.

---

## 9. Nearby Offices Panel

- `publish_nearby_offices` accepts an office list and renders nearby-office cards over `ui.nearby_offices`.
- Used in conjunction with location data to show contextually relevant offices.
- `publish_office_details` renders ONE specific office in detail (with image) over `ui.office_details` — used when the user picks or names a single office.

---

## 10. Contact Form (Voice-Guided Submission)

- Two-step flow: **preview** → **submit**.
- `preview_contact_form` collects `name`, `email`, `phone`, `details` and renders a form card on the frontend (`ui.contact_form`).
- User confirms by voice → `submit_contact_form` fires.
- On submit: email receipt sent via SMTP; submit-status packet pushed to frontend.

---

## 11. Job Application (Voice-Guided Submission)

- Same two-step preview/submit pattern as contact form.
- `preview_job_application` renders job application card (`ui.job_application`).
- `submit_job_application` sends an email receipt via SMTP and publishes submit status.

---

## 12. Meeting / Calendar Invite

- `preview_meeting_invite` collects `recipient_email`, `subject`, `description`, `location`, `start_time_iso`, optional `duration_hours`.
- Renders a meeting form card on frontend (`ui.meeting_form`).
- `schedule_meeting` sends an ICS calendar invite via SMTP email.
- Frontend receives a sent-status packet on success.

---

## 13. Browser Geolocation Request

- `request_user_location` publishes a location request packet over `ui.location_request`.
- Waits asynchronously for the frontend to respond on `user.location` topic.
- Handles three outcomes: `success` (coordinates stored), `denied`, `unsupported`.
- On success: reverse geocode via Google Maps API to get human-readable place name.

---

## 14. Directions & Distance Calculation

- `calculate_distance_to_destination` accepts a `destination` and optional `origin_place` + `travel_mode`.
- **Path A (place name):** geocodes `origin_place` → Google Routes API call.
- **Path B (GPS):** uses stored coordinates from a prior `request_user_location` call.
- Publishes a map polyline to frontend over `ui.location_request` type `map.polyline`.
- Returns spoken distance + duration summary to the LLM.

---

## 15. Context Delivery via Email

- `send_context_email` resolves the current or a prior flashcard snapshot (`screens_back` parameter).
- Composes a human-readable summary via LLM (falls back to raw format if LLM fails).
- Sends via SMTP to the provided or previously collected `recipient_email`.
- Publishes delivery status packet over `ui.email_delivery`.
- Falls back to Mem0 recall if no session snapshot available and `user_id` is known.

---

## 16. Context Delivery via WhatsApp

- `send_context_whatsapp` does the same summary → delivery flow but sends via Meta Graph API using a WhatsApp template message.
- Accepts `recipient_phone` and optional `screens_back`.
- Publishes delivery status packet over `ui.whatsapp_delivery`.
- Handles template/auth/API errors explicitly with failure strings.

---

## 17. User Identity Sync

- `get_user_info` collects `user_name`, optional `user_email`, `user_phone`.
- Pushes a `user.details` packet to the frontend so the UI can display/pre-fill identity.
- Frontend also sends a `user.context` data packet on join; agent ingests `user_id`, `user_name`, `user_email`, `user_phone` and stores in agent state.

---

## 18. UI History Log

- `get_ui_history` returns a server-side newline list of everything published to the frontend in the current session.
- Useful for agent to reference what the user has already seen.

---

## 19. Graceful Call End

- `end_call` shuts down `AgentSession` cleanly via `context.session.shutdown()`.
- Returns a fixed farewell instruction so the LLM speaks a closing line before disconnecting.
- Room is deleted on close (`delete_room_on_close=True`).

---

## 20. Authentication System

Two login methods, two roles:

| Method | Role logic |
|---|---|
| Email + password (`POST /auth/login`) | Role stored in MongoDB per user |
| Google OAuth (`GET /auth/google` → `/auth/google/callback`) | `@ADMIN_DOMAIN` → admin; any other → client |

**Role behaviour:**

| Role | JWT expiry | Notes |
|---|---|---|
| `admin` | 30 days | No access window restriction |
| `client` | 4 hours | Access window enforced from first login (`CLIENT_ACCESS_WINDOW_HOURS`) |

- JWT stored in `auth_session` httpOnly cookie by Next.js.
- Google OAuth callback lands on FastAPI, issues JWT, redirects to Next.js with `?token=...&role=...&expires_at=...`.
- `POST /auth/logout` — returns `{ "success": true }`; cookie deletion handled by Next.js.
- FastAPI route protection via `get_current_user` / `require_admin` dependency injection.
- Client access window reset by unsetting `first_login_at` in MongoDB.

---

## 21. LiveKit Token Issuance

- `GET /api/getToken` — single HTTP endpoint the frontend calls to start a session.
- Creates a LiveKit room if no `room` param provided.
- Dispatches the `indusnet` agent worker to the room.
- Returns a signed LiveKit JWT as plain text.
- Accepts `user_id` (required), `name`, `email`, `agent`, `room` query params.

---

## 22. Real-Time Data Packet Bus

All UI updates flow through LiveKit data channel topics — no HTTP polling.

**Agent → Frontend topics:** `ui.flashcard`, `ui.contact_form`, `ui.job_application`, `ui.meeting_form`, `ui.location_request`, `ui.global_presense`, `ui.nearby_offices`, `ui.office_details`, `ui.email_delivery`, `ui.whatsapp_delivery`, `user.details`

**Frontend → Agent topics:** `user.context`, `user.location`, `ui.context`

---

## 23. Dual-Process Deployment

- **FastAPI process** (`src/api/main.py`) — HTTP API, auth, token issuance, MongoDB init.
- **Agent worker process** (`src/agents/session.py`) — LiveKit agent, all voice/tool logic.
- Both run independently; deployable as separate Docker containers via `docker-compose.yml`.
- `run_both.sh` starts both locally with a dependency preflight and auto-recovery (`uv sync --frozen`).

---

## 24. Rotating Log System

- `logs/app.log` — rotating file handler via `src/core/logger.py`.
- `setup_logging()` called once at startup in both processes.
- All tool failures, delivery errors, filler debug lines, and participant events logged.

---

## 25. MkDocs Documentation Site

- Full architecture documentation under `docs/` — architecture diagrams, auth flows, data flows, tools reference.
- `mkdocs build` produces a static site served at `/documentation` by the FastAPI process when `site/` exists.
- Material theme with Mermaid diagram rendering, code copy, search, and navigation tabs.
